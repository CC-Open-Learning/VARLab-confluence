VARLab : Using the File Share

Created by Jordan Robinson, last modified on Aug 24, 2021

For documents and source files that we don’t need in our sim project repositories, we use a normal network share in order to store and organize our files. This may include files such as storyboarding documents, asset source files, reference images, meeting recordings, and anything else that is not directly necessary to be distributed in the project repo.

Unlike the files in our Bitbucket repositories, content added to the file share is not directly downloaded to our device until we need to open it. Instead we can see the files and their thumbnails but not download gigabytes of files that are only relevant to other modules.

Sharepoint & OneDrive

Our file share service at the moment is a shared drive on Microsoft Sharepoint. While you can upload and browse files online in Sharepoint it is much simpler to use OneDrive on your PC to synchronize a folder with the contents of our shared drive.

To set up OneDrive to synchronize with a folder on your PC, visit the shared drive in your web browser and select Sync from the navigation bar.

When prompted to open the URI with a specific program, choose Microsoft OneDrive and click Open Link.

Once you have logged into OneDrive on your PC you will be prompted to choose a folder to use to hold all of the content in the shared drive.

Organization

In order for our file share to be easy to navigate and keep our content simple to find, we are using a specific file structure for all projects as well as a naming convention for our assets when it is relevant.

While older modules may not adhere to these conventions, we strive to do so for all current and future modules.

Structure

To keep content organized for each of our projects, we are currently using this file structure for our sim projects:

  • LSM#.## Project Name - Subproject - Official name of the module.

    • Assets - Source files for all art assets made for the project.

      • Audio - Samples used for layering, DAW project files, etc.

      • Visual - 3DSMax, Maya, Photoshop, Substance, etc source files.

    • Recordings - Demos and other recorded video material.

    • References - Contains reference materials found online or provided by stakeholders.

      • Documents

      • Photos

      • Videos

    • Planning - For storyboards, Gantt charts, requirements, etc.

    • UAT - User testing documents specific to this module.

Naming Convention

Microsoft Sharepoint is a version-controlled file share. This means that you can safely save over top of files in the file share without having to worry that you’ve permanently lost data. In the event we need to roll back to a previously known good version of a file, we can do that. This prevents us succumbing to bad practices such as naming files file_v1, file_v2, file_v2_final, file_v2_final_seriously, and so forth.

Give your subfolders and files descriptive names so that anyone who needs to find your file has the best chance of doing so in the shortest period of time without needing to ask.

Shared Assets

For assets that don’t quite fit a specific project such as an asset pack, CORE Engine logos, brand materials, etc there is a folder labeled Shared Assets within the file share to store them.

Browsing

While the normal file explorer may be enough for the majority of our team members, to enhance the browsing experience for specific file types there are a few external programs that can be useful.

Images

For browsing images, depending on the preferences of the user some suggestions are:

  • Adobe Bridge or Lightroom

    • Great solution for anyone using the Adobe CC tech stack.

    • Team members who are still actively students at Conestoga have access to Adobe CC, allowing them to use these programs free of charge.

  • XnView

  • IfranView
