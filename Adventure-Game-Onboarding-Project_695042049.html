Created by David Inglis, last modified by Anzhelika Kostyuk on Sept 07, 2023

The purpose of onboarding is for new developers to get familiar with the tech stack that will be used in most of the development work at the VARLab - Unity!

To help you get comfortable with Unity, and the flow of work at VARLab, weâ€™ll use a completed Unity tutorial game which can be built upon to learn new techniques for Unity development. The Adventure Game is a point-and-click adventure set in a cyberpunk city, and the associated tutorial describes a series of systems which make the game work.

This document will cover:

  • setting up the Adventure Game project repository for onboarding

  • planning goals for a week of development in the Adventure Game project

  • where to record notes about your work on the Adventure Game project

  • when and where to save your work (code) from Adventure Game

Getting Started

The repository for this base game is located at https://bitbucket.org/VARLab/unity-adventure-game/src/main/ and uses Unity 2021.3.24f1.

Install Unity from the link above, then follow the steps outlined in the Setting Up Bitbucket & Sourcetree document to get your development environment ready for new projects.

Create a Workspace in Bitbucket in order to create new repositories.

Fork the Adventure Game repository in Bitbucket to create a copy of the repository in your Bitbucket workspace, then use Sourcetree (or a git client of your choice) to clone your new repository to get started.

Before attempting to develop any new features, play through the game a few times. Open the Persistent Scene located in the Scenes folder, then press Play to begin playing the Adventure Game.

Saving Your Work

Throughout your week of onboarding and development in Adventure Game, you should be committing and pushing your work after each meaningful change.

There are two key benefits to ensuring your work is always committed and pushed to Bitbucket:

  1. Work can be restored if your computer crashes, or if you have to switch machines throughout development.

  2. You can roll back to a previous working version of your project if something breaks or goes wrong.

The document on Using Bitbucket & Sourcetree provides steps on using Sourcetree to keep work synced with Bitbucket.

Documenting Your Progress

Each team member has their own personal space in Confluence. For onboarding, whether with the Adventure Game, Roll-A-Ball, or any other tutorial you may work through, use Confluence to document your goals, progress, and setbacks. This information will be the basis for a great demonstration of your work at the end of the week.
