VARLab : Managing Jira Tickets and Workflow in DLX Production Projects

Created by Stacey Dineen, last modified on Aug 10, 2023

The thing to be really clear about is the handoff from Art to Dev (and sometimes back again, when bugs are reported that require art changes). The trick to keeping things from falling through the cracks is to LINK and COMMUNICATE.

Asset Ticket Integration Process

STEP 1. ARTIST OR DEVELOPER - Create 2D or 3D asset ticket and link it to an appropriate dev ticket where that asset will be implemented in Unity. The dev ticket is usually a user story such as “As a learner, I need to plug the power cable into the wall socket so that I can use the machine”. In this example, there could be several different asset tickets linked to the user story, such as the Power cable, Electrical outlet, Electrical outlet cover plate.

EVERY 2D or 3D asset ticket must be linked to a User Story ticket on the development side of the project. If there is no user story for a particular art asset, one of two things must be true: Either the asset is not actually needed in the sim (low priority or non-interactive asset) or the user story is missing! Do not begin work on any asset ticket that is missing a User Story link!

STEP 2. ARTIST - Change the status of the 2D or 3D Asset ticket as needed on the Active Sprint art board, as you create the required asset. When the artwork is completed, you must

  • Upload the asset to SharePoint.

  • Add a screencap and comment in the Asset ticket, along with a direct link to the asset file(s) in SharePoint.

  • Check the linked User Story - if it is in the current dev sprint, make sure you notify the dev team at standup.

  • Change the status of the 2D or 3D Asset ticket to “Awaiting Import to Unity”

Please note: The “Awaiting Import” status should only be used when an asset is 100% finished. This means that no further work is planned or needed for the ticket. Do not request for assets to be imported to Unity if more work is required, unless the remaining work is actually going to be done in a different ticket. Do not continue working on any ticket while it is in in the Import column of the board - if you discover that more work is required, pull the ticket out of “Awaiting Import” status back to whichever other step of the workflow is appropriate (e.g. Texturing). You can always put it back into the Import column later, when the work is done.

STEP 3. DEVELOPER - Every day at standup, look for any tickets on the Art board that are in the “Import” column so that the assets can be imported to Unity. Two possible situations:

  1. The Asset ticket is linked to a User Story that is still in the backlog. Dev is not ready to implement this story yet, but we still need to import the assets now. Create a ticket “As a developer, I need to import the _____________ asset (say what it is - doorknob, fire extinguisher, power cable) so that I can implement it in a future sprint”. LINK the import ticket to (all of) the Asset ticket(s) that need to be imported. Check with the artists to make sure you are getting the right Asset tickets.

  2. The Asset ticket is linked to a User Story that is in the current sprint. Make sure that the owner of the dev ticket knows the asset(s) are ready, and make sure that the User Story includes importing AND implementing the asset(s) in the sim.

STEP 4. DEVELOPER - As soon as your User Story from Step 3 is merged successfully, it will automatically go to status Dev Complete (through the magic of Bitbucket integration with Jira). HOWEVER, the Asset ticket(s) that are associated with that story will NOT be updated automatically. You must manually update the status of all the imported Asset tickets to “Done” when they have been successfully imported and merged into dev.

  • Add a comment in each Asset ticket to reference the merge event in the linked User Story

  • Manually update the status to Done for the Asset tickets only. Do not update the User Story on the dev board - this is done by the project lead when the build is successful.

Remember, it is a RISKY MOVE to manually update the status of any ticket to Done, so BE CAREFUL and make sure you have the correct 2D or 3D Asset ticket(s). Double-check the links, talk to the artists, and get this right! Your name is going all over this in the activity log, so it is worth paying attention and being very sure.

WE NEVER MANUALLY UPDATE THE STATUS OF DEV ISSUES TO “DONE”. If you are tempted to manually move a dev ticket to an inactive status such as Dev Complete or Done, please stop and reach out to Stacey, Sean, or David. NEVER EVER do this, even if you think the status is incorrect and you’re trying to correct it. Talk to one of us and we’ll fix it together. This applies to user stories, bugs, or any ticket that requires code changes to complete the ticket. The only way that these tickets can be moved to status “Dev Complete” is when they are tagged in a PR and merged successfully (Bitbucket pushes the status change to Jira as part of the Merge event). For this reason, it should never be necessary to manually change the status of any dev ticket to “Dev Complete” or “Done”, because the Merge event will take care of this automatically.

It is totally normal and permitted to update any ticket manually between “active” statuses, either forward or backward through the workflow. For example, you can change the status of a ticket in any of the following ways, and that’s completely fine:
To Do → In Progress
In Progress → Review
Geometry → Structure
Texturing → Polish
** OR BACKWARDS when more work is needed **
Review → In Progress (going back to do more work)
Dev Complete → To Do (going back to do more work, but not in this sprint)
Import → Texturing (to fix a texture problem prior to import)

The only forbidden case is when you move a dev ticket from ANY other status → Dev Complete or Done. Never do that - only a Jira Admin such as David, Stacey, or Sean can manually change the status of an active production project DLX ticket to Dev Complete or Done.

At the end of a sprint, when the build is successful and the sprint is completed, the project lead will manually push the status of “Dev Complete” tickets to “Done”. Why is that allowed? Because the tickets are already in a resolved state of Dev Complete. They made the jump from active to inactive when they went to Dev Complete (this is an automatic change, not manual). Updating them to Done when the build is complete is a good way to know when the tickets are included in the most recent demo build.

Asset tickets must be manually updated from Import → Done because they will never get a Merge event trigger from Bitbucket to move them automatically. Asset tickets are ONLY updated to Done when they are linked to a dev ticket that has been successfully merged (status of the linked dev ticket must be Dev Complete or Done). NEVER update the status of any Asset ticket to Done if it is not currently in status Import AND shows a linked dev ticket that is merged.
