VARLab : Art Asset Review Process

Created by Jordan Robinson, last modified on Nov 02, 2021

In this article we will cover the review process we use for art assets to reduce the probability that we will need to return to a previous step and potentially lose work that has been completed during later steps.

Tickets

Artists must find their relevant asset ticket within the board and if one is not present, create it. The title of any created tickets should explain why the work is important for the students that will be using this simulation.

A ticket such as “As a student I need to see air hoses so I can properly inspect for leaks” is a meaningful way to express that the air hoses are important for the inspection.

Review Process

When an artist completes any of the Workflow Steps outlined below, they need to flag the Jira ticket for their asset. This flag will cause the ticket to highlight within the Assets board and also email each of the art leads.

When placing your ticket into review, you will need to include the asset files for review. To do so we’ll be using our file share as outlined in Using the File Share. Place your current source files into the correct folder for the project so that a reviewer can open them and check everything over.

Once their work has been reviewed, an artist may begin working on the next stage of the model. This process is repeated for each step until the model is complete.

Workflow Steps

When an artist begins working on a ticket, they change the status from TO DO to BLOCKOUT or HIGHPOLY depending on the situation.

Large assets with many pieces included that require specific sizing, dimensions, and layout should first go through the BLOCKOUT stage so that the basic shapes and positions of each object can be solidified before any detailed modelling occurs.

Assets of lower complexity can skip to HIGHPOLY as a blockout stage is not necessary for models such as a coffee cup.

White Box (Blockout)

This is a preliminary step before starting to add details for the Highpoly model. In this step the general shape and dimensions of the model are laid out to make sure pieces will fit together as they should before handing off individual parts of the model to different artists.

This step is very important for large models that will be worked on by multiple artists such as the Backhoe model from Heavy Equipment.

Highpoly Model

The first step in the process is to create our high-detail or Highpoly model. This model should be our most detailed representation of the object we are making.

As the highpoly is the first step in the process there are not many special considerations to make during this step.

Lowpoly Model

The Lowpoly model is the lower detail, game-ready model to be used within Unity. This model needs to have the same general shape and silhouette as the highpoly but without the high-frequency details such as rounded corners, nuts and bolts, etc.

While making the Lowpoly we must be careful not to change the silhouette of any items as they need to match up to our highpoly. If we make a change to the lowpoly that is not also done on the highpoly, we will introduce baking errors.

Rigging

Before integrating the model into Unity, the artist must discuss with the development team how the model should be structured. This includes making sure each individual mesh object within the model is named what they’d like and that the hierarchy of all the objects matches what the developers need.

If the model requires skeletal animations, the bone rig and names will also be a part of this step.

Animation

If the model requires animations, they need to be done during this step. For Unity specifically, animations that are not done solely through code are very fragile and can be broken if developers rename your meshes or change their hierarchy once imported into Unity.

Make sure both parties understand this limitation otherwise major issues can arise later on!

Integration

When the lowpoly model is ready, it has been prepared as the developers require it from the rigging step, and animations have been completed, we can now allow developers to use our model within the game engine while we continue on the other steps in the modelling process.

Once this model has been exported for use by the developers, we cannot edit the names or hierarchy of any object in our model. Doing so will cause the developers to lose their work if they have already started to use the model.

We are also unable to change the location of pivot points or move objects within our modelling program without causing issues for the developers upon reimport.

Adding new objects to an existing lowpoly after integration is not as destructive but still not recommended.

UV Mapping

With the model separation, naming and hierarchy finalized, we can begin creating Texture Coordinates for the model by UV Mapping it.

If the Lowpoly model changes, it is most likely that a change will need made to the UV map afterwards. For this reason, make absolutely sure the lowpoly meets all requirements before starting your UV map.

Texturing

With the UV map completed, the next step is to create textures for the model to be used in the game engine.

Baking

Using the highpoly and UV-mapped lowpoly we can bake the details of the highpoly onto the lowpoly model to convert the details of the highpoly model into texture maps for the lowpoly.

Small changes can be made to the highpoly during this step to add additional details. As long as the general shape stays the same, adding things such as additional screws, zippers, etc to the highpoly is not going to cause any problems.

Substance

After the model has a proper UV map, the texture for it can be created. If using Substance Painter, to properly make use of the Geometry Mask feature it is recommended you make a copy of the lowpoly and separate parts of the model out into different objects before importing.

When exporting your materials for CORE, we use a shader that takes the following texture maps as inputs:

  • {Model}_Base

    • The base texture for the model, also known as Albedo or Diffuse.

  • {Model}_MRO

    • This is a Channel Packed texture where each color channel is as follows:

      • Metallic (Red)

      • Roughness (Green)

      • Occlusion (Blue)

        • If using Substance that this is set to your Converted Ambient Occlusion.

  • {Model}_Normal

    • The Normal map of our model, make sure if using Substance that this is set to your Converted Normal (OpenGL).

Level of Detail Models

With the lowpoly, UV, and texture done it is a good idea to place the texture on the model in your modelling program and begin reducing the detail of the lowpoly in steps. These reduced versions of the lowpoly are known as Level of Detail (LOD) models and will be used as the model geometry as the camera gets further from the asset in the game engine.

This step also includes the creation of a Physics Model for the mesh. If the mesh is complex and cannot be adequately represented by a primitive collider (box, sphere, capsule), it will need a low detail mesh to represent it within the physics engine. This will help provide more accurate mouse-based interactions and collisions.

The physics model should be the bare minimum required to represent the shape of the mesh to the physics engine. This model is invisible to the player and thus can be as ugly as you need it to be. A good rule of thumb is that it’s one detail level worse than your lowest LOD model.
