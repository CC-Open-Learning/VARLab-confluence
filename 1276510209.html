Section 1: What is a GLB (glTF)?

  • glTF (GL Transmission Format) is used for the transmission and loading of 3D models in web and native applications

  • The format reduces 3D model file sizes and reduces the processing load to unpack and render each one

  • Commonly used in Web Applications, and is used in Game Engines like Unity, Unreal, Godot, and others

Section 2: Blender glTF/glb IMPORT/EXPORT Features

  • Meshes

  • Materials

    • Principled BSDF (PBR Art Style)

    • Shadeless (Unlit)

  • Textures (used in Materials)

  • Cameras

  • Punctual Lights

    • Point Light

    • Spot Light

    • Directional Light (Sun)

  • Animations

    • Keyframe

    • Shape Key

    • Skinning

Section 3: Meshes

  • glTF’s Internal Structure mimics the memory buffers commonly used by graphics chips when rendering in real-time

  • These assets can then be delivered to desktop, web, or mobile clients and can then be displayed with minimal processing - this results in Quads and NGons automatically being converted to triangles when exported

  • Discontinuous UVs and Flat-Shaded Edges may result in moderately higher vertex counts in glTF vs Blender, as these vertices are separated on export

  • Curves and other non-mesh data are also not preserved, meaning these must be converted to meshes prior to export

Section 4: Materials

  • The core glTF/glb format supports a Metal/Rough PBR workflow with the following Channel Input setup:

    • Base Color (RGB)

    • Metallic (Non Color)

    • Roughness (Non Color)

    • Baked Ambient Occlusion (Non Color)

    • Normal Map (Non Color)

    • Emissive (pulled from Base Color)

  • The glTF/GLB Material System is different from Blenders system, so the Blender system attempts to construct a similar replicated material based on the glTF/GLB files material (as closely as possible).

    • This difference is normal as no 2 applications share the same Render setup (Blenders, Unity, Maya, etc are all different from eachother)

  • Some variation is to be expected → although if needed you can save the imported Texture Maps of the Materials and then reconstruct the materials in Blenders Node Editor

    • As the Textures are contained within the GLB export, locating them in Blender and saving to your computer is a good way to reconstruct the materials

  • Blenders glTF/GLB Importer supports Metal/Rough PBR (Core/Default), Specular/Gloss PBR, and Shadeless Unlit materials

    • Spec/Gloss PBR Extension: KHR_materials_pbrSpecularGlossiness

    • Shadeless Unlit Extension: KHR_materials_unlit

  • Blenders glTF/GLB Exporter supports Metal/Rough PBR (Core/Default) and Shadeless Unlit materials

    • Shadeless Unlit Extension: KHR_materials_unlit

  • When Exporting, Blenders Exporter constructs a glTF/GLB material based on the Material Node setup that the user created and the related information that it is able to recognize from these

    • NOTE: The glTF/GLB materials require either a .PNG or .JPEG images be used → If a different type is being used then Blender will automatically convert the files to the required types, but at the cost of an increased Export process time

  • For creating Shadeless (Unlit) Materials, use the Background Material Type

    • Material Tab → Surface → Dropdown → Background

    • Using this Setting setup will engage the KHR_materials_unlit Extension

Section 5: Material Texture Specifics

Below are notes for each specific Texture Map Type and how to best utilize it for the glTF/GLB format.

Production Note: Technically speaking, each texture map below can be utilized with the Principled BSDF as separate maps (ie: 1x _color, 1x _normal, 1x _ao, 1x _metal, 1x _rough, and so on). This is important and useful for things like Tile Textures (which often come packaged as individual maps) and means we don’t need to go through the hassle of combing maps and generating things like _MS, _ORM, etc

Base Color Texture Notes

  • Self explanatory - uses the color map socketed to the Color Input of a Principled BSDF Node

  • If there is no Texture Map socketed to the Principled BSDF Color Input, then the Principled BSDF’s Color Selector will be use for the input (Default = White)

  • For any Texture Map being use for color, ensure that the RGB Color Space is used

Metallic and Roughness Texture Notes

  • Read from the Principled BSDF Node Inputs of the same name (Metallic and Roughness)

    • Ensure that any Metallic, Roughness, or _MS textures used are set to the Non-Color Space

    • These maps can be used individually on their respective nodes, or a map combining the Metallic and Roughness can be used

      • The corresponding textures that the Lab uses are _MS (Metallic Smooth)

    • For _MS setups

      • The Metallic Map is read from the Blue Channel and the Roughness Map is read from the Green Channel

      • To Separate the Channels of the _MS map → Use a Separate RGB Node to split the channels and hook the Blue and Green channels up to their respective inputs

    • If No Texture Maps are socketed into the Principled BSDF Metallic and/or Roughness Node Inputs, then a slider with a range of 0 → 1 can be used to adjust the Metallic and Roughness appearance

      • For Roughness, a value of 0 will result in a very glossy appearance to the material, while a full 1 will result in a very flat and opaque appearance

    • When Exporting, if the Metallic/Roughness/_MS maps are incorrectly hooked up via the Separate RGB Node, then Blender will attempt to export a converted setup, at a cost of an increase to the Export processing time

Production Note: We are looking at implementing an additional ORM texture export into our Substance Painter Template so as to combine our AO, Metallic, and Roughness maps and eliminate the need to use an _MS texture and an _AO texture separately

Ambient Occlusion Texture Notes

  • Generally, AO Texture Maps can be used in a Principled BSDF Node Setup by using a Mix Node set to Color, and set to Multiply with the Base Color map being plugged into the A Input and the AO map plugged into the B Input. The output of the Mix node can then be plugged into the Principled BSDF Color Input

    • An alternative method is to use a Custom glTF Settings Node Group to store and output the AO information

      • To Create the Node Group:

        • 1: Add “AO Node”

        • 2: Right Click on the Node

        • 3: Click “Create Group”

        • 4: Open Tool Panel

        • 5: Remove ALL Outputs and remove ALL Inputs, except for Color

        • 6: Rename the Color Input to “Occlusion”

    • glTF stores the AO information in the Red Channel, allowing for Optional Sharing with the Metallic (Blue) and Roughness (Green) Channels

      • This is known as a ORM Map (RGB = AO[R], Rough [G], Metal [B])

    • If an existing AO Map exists then it can instead be hooked directly into the custom glTF Settings Node Group and be read from there

    • There is no true way to fully bridge AO functionality in Blender for transfer to glTF/GLB, but this Custom Node Group setup achieves the bridge in most cases

    • If the Blender Exporter finds a Custom Node Group named “glTF Settings” with an input named “Occlusion” on that Node Group, it will look for an attached Image Texture → This will write a linked Image Map (_AO) from the main Material Tab to the Export

      • This means that the AO Map and glTF Node do not to be linked to the Material Output Node

Normal Map Texture Notes

  • Normal Maps are pretty straightforward when it comes to the glTF/GLB format and is much the same as other formats

    • Connect the Normal Map Image Node to a Normal Map Node, and then feed the Normal Map Node into a Principled BSDF’s Normal Input, making sure to set the Normal Map Image Node’s Color Space to Non-Color

    • One exception however is that the Normal Map Node must be set to Tangent Space, as this is the only type of Normal Map supported by glTF/GLB currently

Emissive Materials Notes

  • Word of Caution: Be careful with how high you set your Emission Strength - the glTF/GLB format can often over-amplify the emission strength of a material and lead to a surface being far more emissive than expected.

    • General Ratio: 0.1 Emission in GLB = 1.0 Emission in Blender

    • A surface can be made emissive by hooking the Color Map of a Material up to the Emission Color Input of a Principled BSDF Node. The Emission Strength Value can then be adjusted to a desired strength

    • Alternatively, an Emission Node Output can be mixed with the Output from a Principled BSDF Node using a Mix Shader Node or an Add Shader Node → With the Emission Node’s Output hooked into the Top Input of the Mix Shader Node and the Principled BSDF in the Bottom Input

    • It is often a good idea to have a material slot of a Model dedicated to Emissions (to give better control over the appearance of the surface without affecting the rest of the model)

Alpha/Transparent Material Notes

  • By making areas of a Color Map transparent in Image Editing (PhotoShop, paint.net, etc) allows for the Color Map to be used to store the Alpha/Transparent information that can then be used in an Alpha Material in Blender (and other softwares) to make surfaces Transparent

    • In an Alpha Materials settings in Blender, adjusting the Blend Modes property in the Materials Tab → Settings Options controls how the Alpha/Transparency information of a texture is read and used

      • Opaque: Alpha values ignored (on by default)

      • Alpha Blend: Blends the transparencies of other elements

      • Mask (Alpha Clip): Transparent values below the set threshold will not render those areas - useful for elements such as leaves or holes through cloth

    • When using the Alpha Blend mode, it is important to only use a single layer of Alpha surface on a mesh (ie: 1 plane of glass over an Arcade machine screen, and no others) and avoid layering any further elements on the given mesh

      • Only 1 Alpha Surface Layer over an Opaque Surface on any given mesh

Back Face Culling / Double Sided Rendering

  • To optimize a Meshes Surface for rendering, the Back Face Culling option should be enabled in the given Materials Settings options

    • For materials with an Alpha layer that needs to be viewed from both sides (such as a character card/plane) you should disable this

      • Do so sparingly as this can affect rendering performance

Section 6: Notes on Lighting

  • As noted above, there are 3 light types in Blender that are glTF/GLB Compatible when exported

    • Point Lights

    • Spot Lights

    • Directional (Sun) Lights

  • Area Lights are not supported by the format and therefore cannot be exported as part of setups

  • As these are Real Time lights, it is important to remember that they are expensive on performance and should be used sparingly. A handful of lights will likely suffice, however its likely that after 20+ are added to a scene a heavy toll will be taken on the performance overall

  • glTF/GLB generally will overamplify lights that are exported and lead to un-intended lighting errors when implemented in a scene. Generally speaking, the ratio appears to be 1.0 Strength in Blender = 10.0 Strength in glTF/GLB.

    • If you desire a light to be 1W in glTF/GLB format, then a strength of 0.1 W (or 100mW) should be used in Blender

    • Also ensure that you utilize Blending options to ensure that lighting is more even when rendered

Visual Reference of what strength settings in Blender will likely appear as in Editors

Section 7: Notes on Animations

  • The glTF/GLB Format allows multiple animations per file, with animations targeted to particular objects at time of export

  • To include animations, either:

    • A) Make the action the active selection on the object

    • B) Combine the animations onto a Single NLA Strip Action

    • C) Stash the Actions

  • Only certain types of animations are supported:

    • Keyframes: Translation, Rotation, Scale

    • Shape Keys

    • Armatures/Rigging/Skinning

  • Animations can be exported in several ways based upon the Animation → Mode setting used in the Export Options panel

    • Actions (Default option)

      • Animations recorded as Actions can be exported either by

        • A) When needing a single action animation, making the desired track the Active Action on the Object and removing all other actions, or

        • B) All associated actions are stored within the Active Object by Stashing them inside it

        • Actions not associated with an object for export, as described above, are not exported so be mindful

      • To have multiple objects of an export that share a similar animation be played as a single action, ensure that the associated Actions share the exact same name (this will play them both as a combined action, rather than separately)

    • Active Actions merged

      • In this mode the NLA organization is not preserved and a single combined animation is exported using the active actions of all objects selected for export

    • NLA Tracks

      • Each NLA Track will be exported as an independent glTF animation

      • Useful if using Strip modifiers or have have multiple actions combined on a single track

      • To have multiple objects of an export that share a similar animation be played as a single action, ensure that the associated Actions share the exact same name (this will play them both as a combined action, rather than separately)

    • Scene

      • Using the Scene Option will export animations as you can see them in the Viewport - and can choose between exporting a single combined glTF animation, or each object separately

      • As previously mentioned, only certain types of animations are supported: Object Transforms, Rigged Animations, and Shape Key Values. This means that animations of other properties like physics, lights, or materials will be ignored

      • Note: To sample shape key animations controlled by bone transform drivers, the animations must be on a direct child mesh object of the rig/armature being exported
