VARLab : Using Bitbucket & Sourcetree

Created by David Inglis, last modified on Sept 01, 2022

In this article we’ll cover how to use Bitbucket and Sourcetree after you have them installed.

For information on how to install and set up your Bitbucket and Sourcetree, check out Setting Up Bitbucket & Sourcetree.

Standard Git Workflow

Creating a branch from a Jira issue

A new branch can be made directly from a Jira issue either on a Jira board or from Bitbucket.

  1. Select a :task: Task, :story: Story, :epic: Epic, or Bug from a Jira board or the Jira Issues tab on Bitbucket

  2. Click Create branch to open a new tab on Bitbucket and create a branch

  3. On the From branch dropdown, select the main branch for the project, typically main or development

    • If you would like to branch off of other work currently in progress, you can choose the branch for that work instead.

  4. The Branch name field will automatically be filled out with the ticket ID and its title

    • Change this name to contain the name of the project you’re working on followed by the ticked ID and a brief set of keywords to describe what the branch is for

      • In this example, we will name it motivepower/MP-230-Task-Grouping

  5. See the section below on Checking out a branch to switch to the newly created branch

Checking out a branch

  1. Click the Fetch button along the top menu bar of Sourcetree to get updates from the Bitbucket repository

  2. It is best to have a clean working directory before checking out a new branch. Any file changes should either be Commited (to save those changes to the current branch), Stashed (to temporarily save the files locally, which can be retrieved later), or Discarded (to permanently delete any untracked changes).

  3. If the branch you want to check out is listed under the Branches tab on the left sidebar, right-click on the branch and select Checkout <branch name>

  4. If the branch is not listed in the Branches tab, you will have to first create a local copy of the branch. The Remotes tab on the left sidebar of Sourcetree lists all branches of the remote repositories (the origin remote corresponds to the Bitbucket repo). Right-click on the desired branch and select Checkout origin/<branch name> to open the checkout dialog in Sourcetree. Keep the default settings and select Ok to create a local copy of the remote branch

  5. After checking out a branch, the branch name will become bolded in the Branches tab with a circle bullet on the left indicating it is the working branch

  6. The Pull button may show a notification indicating if there are new changes. If there is new code on the remote branch that is not yet in your local copy, you will have to Pull the new changes to your machine. After clicking the button and launching the Pull dialog, ensure that the branch indicated by the Remote branch to pull dropdown matches the Pull into local branch field

Committing and pushing changes

A commit is a checkpoint for one or more file changes, additions, or deletions. A push sends one or more commits from the local repository to the remote Bitbucket repository.

  1. Click the Commit button on the top menu bar of Sourcetree to bring up the Commit dialog

  2. Click the + button beside any files in the Unstaged files window you wish to include. This will add them to the Staged files window, and similarly they can be removed before committing by clicking the - button if a file was unintentionally staged

  3. Leave a message describing the changes in the commit, then click the Commit button at the bottom right below the message window when ready

  4. After creating one or more commits, the changes can be pushed to the remote Bitbucket repository. Clicking the Push button on the Sourcetree menu bar brings up the Push dialog (the default options in the dialog are usually sufficient, as it sets up a push on the working branch from the local to remote copy)

Referencing Jira issues by number in commit messages will automatically create hyperlinks on Bitbucket for those issues when reviewing the commit.


For example: “This fix resolves issue CORE-53, which blocks CORE-56”

Creating a pull request

After working on a feature or bugfix on a dedicated branch, a pull request indicates that the branch should be merged back into development to have the new changes integrated with the workflow of the other team members.

  1. In Sourcetree, right-clicking on a branch under the Branches tab and selecting Create pull request… opens the Bitbucket pull request dialog in a web browser

  2. This dialog can also be reached from Bitbucket by navigating to the Branches tab on the left sidebar, then selecting the branch to be merged. The hamburger menu on the top right of the page hides the Create pull request link

  3. Ensure that the selected branch on the left is the feature/bugfix branch to be merged, and that the selected branch on the right is the development branch

  4. The Title and Description fields will be auto-populated with information from the commit messages for that branch

  5. In the Reviewers field, team members can be referenced who will review the pull request, check out the code, and merge the branch into development if no issues arise

  6. If the branch to be merged resolves the corresponding Jira issue, the Close branch option should be checked. Otherwise, if there is still work to be done on that specific branch, leave it unchecked

  7. Click Create pull request to send a notification to the Reviewers

For all visual changes such as UI layouts, 3D assets, color changes or anything else that requires someone to be able to see it to review it, make sure to include screenshots in your description field for easier viewing.
