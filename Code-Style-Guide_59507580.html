This document outlines the style required for source code and comments for C# and Unity development. These style requirements are based on Conestoga College’s SET Coding Standards and Microsoft C# Coding Conventions.

Commenting structure is also informed by Microsoft .NET XML Documentation which will provide helpful information in Visual Studio via Intellisense, and can be used to generate wiki-like documentation files.

Naming Conventions

C# programs use PascalCase for type names, namespaces, and all public members.

  • Identifiers shouldn't contain two consecutive underscore (_) characters. Those names are reserved for compiler-generated identifiers.

Classes, Methods, and Events

Classes, methods, and event names should be written in PascalCase and have clear names that match their purpose, as suggested by the Microsoft .NET Documentation.

Public Members

Public fields and properties should be written in PascalCase and represent the data they store.

Private Members

Use camelCase when naming private or internal fields.

Function Parameters

Use camelCase when naming method parameters.

Function Variables

Use camelCase when naming variables within functions.

Interfaces

Should be written in PascalCase with the letter I affixed to the beginning of the phrase.

Attributes

Should be written in PascalCase with the word Attribute affixed to the end.

Enums

Should be written in PascalCase using a singular noun for non-flags and a plural noun for flags.

Constants

Constant fields should be written in PascalCase.

Access Modifiers

Classes, methods, members, and properties must always explicitly define their access modifier (public, private, etc) even when assumed to be private by default.

Comments

Comment Tags

Required Tags

The <summary> tag is used to provide a general overview of a field, property, class, or function.

The <param> tag is used to describe a parameter used by a function It is important to include the name of the parameter in the tag (<param name="firstParameter">) as this helps Intellisense tie the description of a parameter to the actual value.

The <returns> tag is used to describe the return value of a function.

Optional Tags

The <remarks> tag is used in a similar manner to the <summary> tag, to provide more information for a class or function.

The <value> tag can be used in place of, or in addition to, the <summary> tag for fields and properties.

The <see> tag is used to link to other resources from within a comment block.

  • class or member reference (<see cref="UnityEngine.Debug" />)

  • URL (<see href="https://example.com">Example</see>)

Header Comments

Classes

Each class should contain a header comment with a description of the functionality provided by the class. All class header comments must provide at least a <summary>. To provide more detail about a class, use <remarks>.

Example

/// <summary>
///     Uses advanced AI to determine how to rewrite code.
/// </summary>
/// <remarks>
///     CodeWriter is dependent on the <see cref="AICodeWriter" /> library.
/// </remarks>
public class CodeWriter { }

Methods

Each method must include a header comment describing the purpose of the method using the <summary> tag, and descriptions of its inputs and outputs using the <param> and <returns> tags, respectively. For methods which have no parameters or a void return type, the respective tags can be left out.

Example

/// <summary>
///     Calculates the total for the provided transactions.
/// </summary>
/// <remarks>
///     First the subtotal is calculated and then multiplied by tax of 10%.
///     The result of the calculation is then added to the subtotal to get 
///     the final total.
/// </remarks>
/// <param name="transactions">
///     A list of transactions for the order of type double
/// </param>
/// <returns> The calculated total for the provided transactions </returns>
public double CalculateTotal(List<double> transactions) { }

In-line Comments

In-line comments are comments placed within a function or method. These comments are used to document the flow of logic within your source code.

In-line comments often serve to inform other developers (or your future self) about important blocks of code, difficult-to-understand or obscure code, or workarounds that were implemented for unexpected problems.

Example

// To get the total, add 10% of the subtotal to the initial subtotal
total = subtotal + subtotal * 0.10;

It can be difficult to understand what makes a valuable comment. In-line comments are the most useful in situations where the outcome of your code is not immediately obvious.

Example

var str = "helloThere";
// Parses "camelCase" as "Camel Case"
str = Regex.Replace(str, "(\\B[A-Z])", " $1").Capitalize();

Without the comment, this code is almost entirely nonsensical to someone without Regex experience. While there is nothing wrong with this comment, it could be removed entirely by moving line 3 to its own function and using a descriptive function name paired with a good <summary> comment.

While it may seem better to comment every line of code written, over-commenting can harm readability. Comments have also been known to lie, especially over time as the codebase changes but the comments do not.

Property Value Comments

If a property represents a value that is not immediately obvious, it can be helpful to add a <value> tag along with the <summary> tag to describe in more detail the value that the property represents.

Example

/// <summary>
///     Number of players currently in the game.
/// </summary>
/// <value> The current value of <see cref="playerList.Count"> </value>
public int NumPlayers { get { return playerList.Count; } }

Unity Serialized Field Comments

Instead of the <summary> tag, use the [Tooltip] attribute for fields that are serialized by Unity, such as public fields, or protected and private fields using the [SerializeField] attribute. The comment within the [Tooltip] attribute will be shown in the Unity Inspector when hovering over that field.

Example

[Tooltip("Value of one dollar.")]
[SerializeField] protected float dollarValue;

Coding Conventions

Member Type Order

When defining fields, properties, and methods in a class, the following list defines the preferred order of each of these member types:

  • Enums

  • Structs

  • Interfaces

  • Classes

  • Fields

  • Delegates

  • Events

  • Properties

  • Indexers

  • Constructors

  • Destructors

  • Methods

Implicit Typed Local Variables

Use the var keyword where their type can be implicitly determined. Do not use implicitly typed variables in loops.

Code Cleanup

Visual Studio provides code cleanup tools to help with automatic formatting. Before committing, run the code cleanup tool by using the keyboard shortcut combination Ctrl+K, Ctrl+E.

This tool can also be configured to run automatically each time a file is saved. Optionally enable this setting by navigating to Tools → Options → Text Editor → Code Cleanup and check the box to “Run Code Cleanup profile on Save.”

Code Cleanup Profiles can be modified for custom style guides, however the default profile is configured using Microsoft’s MSDN style guide and need not be modified.
