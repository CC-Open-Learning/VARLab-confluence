Definitions

  • User Acceptance Testing (UAT) - This is when software is tested in the “real world” by actual users to try and spot things that may have been missed during development, or that could be improved for better user experience.

  • Code Coverage - This is a metric, typically represented as a percentage amount, of how much of your code is covered by tests. The closer to 100% this is, the closer your project is to being fully tested.

What will you learn?

To write effective tests, it is essential to understand exactly what we are and are not trying to test, and why.

If this is not understood, many tests will be ineffective, leading to problems such as a bloated test suite or tests that have been rushed.

It can be easy to fall into the habit of writing tests blindly to reach 100% code coverage. This document is intended to teach you what exactly you should be looking to write a test for when working on a project.

Risk-Based Code Coverage

When writing tests, it is essential you follow the mindset of Risk Based Code Coverage. This means thoroughly testing the riskiest sections of your code, instead of only writing tests to achieve 100% Code Coverage.

This is an example of “Quality over Quantity.” Having ten tests that expose three severe bugs and ensure the most complex sections of code work, but only cover 45% of your code, is more effective than having 100% Code Coverage with thirty tests where you are testing things that have a low chance of breaking.

While you still want to aim for a good Code Coverage percentage, a higher percentage does not equal useful tests. There is a balance between having good Code Coverage and ensuring tests are purposeful and testing meaningful code.

Furthermore, be sure to avoid strict code coverage restrictions within your project. Estimates at certain points in development may be helpful, but you should not have a strict percentage. This might lead developers to write tests purely to hit that goal rather than focus on quality.

User Bugs vs Developer Bugs

It is essential to know the difference between User Bugs and Developer Bugs, and which ones we’ll tackle with testing.

User Bugs

User Bugs are typically found during UAT, usually when we, as developers, expect the user to do one specific thing, and they get lost and do something off course from our expectations. This is NOT what we are aiming to test. It is what UAT is for.

Developer Bugs

Developer bugs are typically gaps in logic in the code. Maybe something was multiplied wrong, or a high score was incorrectly saved. These bugs are what we are hoping to find with our tests. Adding automated tests to this type of logic helps us verify that it works exactly how we intend.

It is crucial to know the difference between the two because developer bugs can still appear in UAT. A developer should be able to recognize the difference between them in this case. Ideally, testing will catch them before ever reaching the UAT stage.

Examples of what to test

The following are some examples of things that should be tested:

Complex code

If you have complex code, you will want to write tests to ensure it works as intended and that there are no gaps in logic that could result in something like a crash.

Complex code could include but is not limited to, loops, nested code and complex calculations.

Error Prone Code

If you have code that could cause errors, this should be tested. You do not want an unexpected crash to appear after the project is live. Check for all possible crashes and ensure you are handling them.

Error-prone code could include anything File IO. Even if you are unsure if the code you have written is error-prone, it might fall under this category.

Complex Getters/Setters

If you have default getters/setters that look like the snippet below, you do not need to test these, since there is nothing to test.

int age { get; set; }
string name { get; set; }

However, if you have getters/setters that look like the following, it may be worthwhile to test it as it does have some semi-complex logic. If this logic fails, it could cause a bug.

 public double finalPrice
        {
            get => finalPrice;

            set
            {
                if (value >= 0)
                {
                  finalPrice = value + (value * 0.13);
                }       
            }

Here the setter of the final price variable includes a calculation. It is worth it to write a single test to ensure it works properly. The get is still the default, so there is no need to test it.

Complex Constructors

Constructors are treated similarly to getters and setters. The more complicated a constructor becomes, the more useful it is to write a test for it.

If you have a constructor like the snippet below, there is no need to write a test for it.

public Person(string name, int age)
{
  this.Name = name;
  this.Age = age;
}

However, if you have a constructor like the following, writing a test for it may be necessary to ensure it works properly.

 public InventoryImpl(
        Action onWin,
        Action onLose,
        IText gemCountText,
        IText countDownTimerText,
        IText currentLifeText)
    {
        OnWinEvent = onWin;
        OnLoseEvent = onLose;

        GemCountText = gemCountText;
        CountDownTimerText = countDownTimerText;
        CurrentLifeText = currentLifeText;
    }

Examples of what not to test

The following are some examples of things that should not be tested. It is important to know when not to test, so that time is not wasted.

Unity Code

We should avoid testing Unity code. We should trust that the code provided by Unity will work correctly, so there is no point to testing it.

As an example, consider the transform.Position function. Unity provides it, and has tested it, so we can trust it works and do not need to test it ourselves.

Microsoft’s libraries (or any other language)

The same rule applies to Microsoft’s native C# functions. For example, we do not need to test the C# String.Equals method, because it is provided to us and already tested.

Trusted Libraries

Lastly, we do not need to test code from trustworthy libraries. Again, it has already been tested and proven to work.

In conclusion, focus on writing tests only for code that you have written, since the code provided by external frameworks and libraries has usually already been tested.
