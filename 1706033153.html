VARLab : Project Intake & Test Request Handling

Created by Ruwini Perera, last modified on Apr 04, 2025

“ Test request handling is crucial to ensure the quality and the reliability of testing .”

Purpose | Earliest Ways of Handling Test Requests | Implemented Solution | 1. Project Intake MS Forms | 2. SharePoint List (Automated) | 3. Dashboard

Purpose

The purpose of this research is to propose a solution to address the issue of not having a centralized platform to store and view the testing requests which often confuses the team.

Earliest Ways of Handling Test Requests

VARLab had no centralized way to store or analyze test requests. Most of the time, teams used MS Teams chats and channels to share this data.

Challenges:

  • Lack of clarity in tracking test requests.

  • Difficulty in retrieving information due to the need to scroll through chat histories.

Implemented Solution

  • MS Forms is used to collect test requests from Principal Investigators.

  • SharePoint List automatically triggers when a request is submitted.

  • A dashboard is populated using data from the SharePoint list, providing clear visibility into all requests.

1. Project Intake MS Forms

This form is designed to collect essential details about new projects testing , including testing scope, release type, required number of resources, and key dependencies and risks. The information gathered via the application will help in test planning, resource allocation, and providing priority to ensure smooth test execution.

Intended Audience : Principle Investigators

When to fill the document ? This document should be completed whenever test planning is taking place and resources need to be allocated for testing.

The fields that need to be filled Includes ; Project Name , Project Description , Release Type (General , Milestone , Sprint Demo ), Number of Resources Requires , Test Duration Estimate ( For how long are you requesting the testing for? ) Key Dependencies & Risks and Additional Comments.

2. SharePoint List (Automated)

When a request is submitted through the list, the SharePoint list is automatically populated with relevant details using Power Automate.

Priority Setting: As part of the automation, an additional column is populated with the Priority of the request. This is determined based on the Release Type selected by the user when submitting the request.

General Release HIGH | Milestone Release MEDUIM | Sprint Demo LOW

3. Dashboard

A dashboard is automatically created using the data populated in the SharePoint list.This dashboard can be customized based on various parameters, such as status and priority.

  • The dashboard can be accessed directly from the SharePoint list, offering a real-time view of the requests with dynamic filtering options based on status and priority.
