VARLab : User Acceptance Testing (UAT) Process

Created by Germ√°n Gaussmann, last modified by Stacey Dineen on Jan 13, 2023

Note that the UAT prorcess was used extensively until VARLab began providing playable end of sprint demo builds to CE’s and project stakeholders. The demo builds are made available to the project stakeholders for them to playtest and provide feedback at the end of every sprint, which reduces the need for a big, formal evaluation at the end of the development phase of the project. For some projects, we will still execute a formal student evaluation of the DLS prior to General Release, but process updates such as incremental release schedules and Service Updates have drastically reduced the pressure to obtain formal pre-release “acceptance” of the DLS.

At the conclusion of each DLS project, prior to submitting the DLS to VARLab Leadership Team for general release, we are required to obtain feedback from real-world users (students and instructors) to validate that our DLS is usable and that it satisfies its learning objectives. This page describes the steps involved to complete that stage of the project.

Ideally, the UAT Process should begin when the project team estimates that there are less than 5 sprints left in the project. At that point, you need to request that the content expert(s) provide a list of approximately 10-15 students and instructor volunteers who can participate in testing.

For externally developed sims, remember to give yourself time to get familiar with the DLS. See UAT- Externally Developed DLS page for more details

Design the UAT Feedback Survey

o   Determine where the user will be intercepted.

o Ask the content experts for particularly important pieces of the simulation. Make sure that we validate with end users and instructors (other than the content expert) that the key learning objectives have been met

o   Think of meaningful questions to ask the users. To be meaningful, the questions need to provide information that can be acted upon. For every question you compose for users, make sure it is clear what you would need to do to correct the situation, if the response to the question indicated that users were unhappy or dissatisfied with the DLS. If you ask “Did you enjoy this simulation?” and the majority of your test volunteers say “No”, you have nothing to go on. You can either ask more specific questions, such as “Do you feel like this simulation provided you with enough information to prepare for your in-person classes?”, or you can ask follow up questions, such as “Please describe what information is missing” or “What changes could we make to provide the right information?”.

  • Create the survey with Microsoft Forms (I like to copy the survey(s) from a recent project and then update them) and add a Collaboration link in the VARLab’s SharePoint folder and/or on the project UAT page. MS Forms will give you (by default) a link that allows someone to respond to your survey, but that is NOT the link we want. In the Share menu, scroll down until you get to the “Share to Collaborate” link - copy THAT link into the SharePoint folder and the UAT page.

  • Add the method to intercept the users in the simulation.

    • This can be a link inside the simulation (which needs to be removed prior to general release)

    • This can be a link added to the UAT course shell, that needs to be clicked after the volunteer has completed the simulation. Both are valid, choose the option that you think is best and will get the best response rate.

  • Get volunteers to test the simulation.

o   Ideally both content experts and end-users.

  • Distribute the version of the simulation with the interception method.

    • There are two different UAT course shells that can be used. Ideally, the simulation that is currently under test will be the ONLY content on the course (remove or mark not visible other content)

    • Option 1: Overview - VARLAB-UAT - MODEL - VARLab User Acceptance Testing (desire2learn.com)

    • Option 2: Table of Contents - VARLab UX Testing (desire2learn.com)

    • Each volunteer must be enrolled in the UAT course shell to be able to access the sim. Only an eConestoga account with “Group Leader” permissions can enroll “students” in the course. If you do not have permissions to access Course Admin > Classlist > Add Participants (including the option to add participants with role = Student) you will need to ask another VARLab staff member for assistance.
      ALL volunteers should be given the role of Student, regardless of their real-life role.

  • Collect the feedback and analyze it.

    • Open bug tickets in Jira for all user feedback. Clearly indicate it is UAT feedback, copy and paste directly from the feedback form in the tester’s own words where possible. Even if the problem cannot be resolved at this time (i.e. it is a limitation of the platform), record the bug anyway.

    • Prioritize the bugs (do not use the default “Medium” priority.

    • Record bugs that you are unable to reproduce. Ask a colleague to try to reproduce. If nobody can reproduce the issue, add comments and mark the bug as ‘Archived’. Do not delete it, it is part of our testing record.

  • Create the UAT report: This is a one-page summary - please feel free to copy this SAMPLE page and fill it in with your own project details. If you choose to create your own report, that’s okay! Just make sure that it contains all of the following elements:

o   Name of DLS project and link to Confluence UAT page

o Recommendation to VLT: Release or Further Development Needed

o   Link to original feedback (e.g. export survey results to Excel, store results in SharePoint project workspace)

o Summary of feedback, focus on usability and learning objectives. See the SAMPLE for ideas.

o   Link to project backlog in Jira (these are the unresolved issues that are known at conclusion of UAT)

  • Submit the report for review.
