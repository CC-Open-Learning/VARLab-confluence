Below is a quick checklist of topics that your peer reviewer will look for when you submit a PR. Please be kind to your reviewer, double check this checklist before creating a PR, and include a brief summary of changes to help guide your reviewer in the right direction. If your changes do not follow anything in this checklist, expect to receive a “Requested Changes”.

Pull Request Formatting

  1. Commit messages get automatically pulled in as the description for your pull request. Save yourself time, and build a good habit of writing good commit messages

    1. The best commit messages describe "why" a change was made, in addition to "what" the change was

  2. Ensure your PR title is a short descriptive summary of the changes

  3. Include screenshots, and comparisons of before and after your changes in the description

  4. Include a brief description to help guide your PR reviewer

  5. Use the tasks feature in Bitbucket as desired to track concrete changes requested during the review process

Scripting

  1. Was anything added along the way during development that isn’t needed anymore?

    1. It can be useful to mark test changes that you are unsure about about with a “//DEBUG” comment, and Ctrl-F through the project before committing

    2. Are there any using statements that are not used? Specifically editor or experimental namespaces accidentally added

    3. Are there any variables/methods that are not used?

    4. Do you have any zombie code blocks or debug logs that should be removed?

  2. Are all naming conventions up to date? Were any variables changed from public to private, etc. and not updated?

    1. Reminder that [SerializedField] variables will lose references in the inspector after name or scope changes

    2. Code Style Guide

    3. Run Code Cleanup in VS before committing

    4. Check over compiler warnings (There should be none)

    5. Check formatting info (Green underlines)

  3. Do classes/methods have more than one responsibility?

    1. Review SOLID principles

    2. Can anything from the class be separated into other classes, helper classes, etc.

    3. Is there any duplicate behaviour between this class and another class?

  4. Are USS classes in scripts that need to be toggled in constants?

    1. USS classes are toggled using .EnableClassInList(“className“)

  5. Make sure there are comments

    1. Methods should have the default function header “///”

    2. Reminder that it doesn’t have to be all or nothing, comment methods and explain why for confusing or complex blocks

  6. Double check that all automated tests pass

Engine

  1. Are the scene changes necessary? The Unity scene is a merge conflict hotspot which should be avoided as much as possible. Always work in the lowest level prefab possible

  2. Do prefabs have any unapplied overrides?

  3. Are the chosen colliders appropriate for the context? Are they optimized for performance?

    1. Are colliders box colliders where possible?

    2. Are mesh colliders marked as “Convex” where possible?

  4. Do your changes need to be a separate prefab? If you are adding objects or components that are significantly complex or if the object is going to be reused then it should be its own prefab

    1. Always work in the prefab you are intending to change

    2. Double check overrides on child/parent prefabs. Sometimes they have the same value but it is still marked as an override

    3. Make sure overrides are intentional in parent prefabs. We might want to override values for scene specific changes that we do not want applied to the child prefab

UI (Toolkit)

  1. Are there any inline styles that should be USS classes?

  2. Is there duplicated USS in multiple classes that can be separated into a shared class?

  3. Do USS classes contain properties that are defined within velcro-structure.uss, or velcro-style.uss and can be applied from those files?

  4. Do USS classes follow the BEM naming convention?

  5. Do USS classes avoid the use of complex selectors (>, #, etc.) where possible?

  6. How does the new piece of UI scale when stretched/shrank? Does it break, and at what point?

    1. Does text wrap, or collapse into “…”, or does it run off the UI?

    2. Does the new piece of UI stay within the safe bounds of your screen? UI should have sensible maximum horizontal values and then proceed to wrap to scale vertically when scaling with text

  7. Does the new piece of UI match the Figma look and feel when in play mode?

Project Files

  1. Are any new files in the correct location?

  2. Are old files that aren’t needed anymore deleted?

    1. Replace existing files as opposed to adding <filename>_new or <filename> (1)

  3. Do new files follow the project’s naming convention?

    1. Some projects have import presets set up, which auto change the texture type as long as it follows the correct naming convention

      1. Ex. "*_Normal" for normal maps, "*_Sprite" for sprites

  4. Replace files in the window file explorer rather than the unity editor

    1. A common workflow is deleting the old one, importing the new one, and renaming the new one to match the old name. This keeps the same identifier in the .meta file

  5. Make sure to only stage files that you meant to change. Always review files instead of clicking the “Stage All” immediately

Art

  1. Are materials created and applied to the fbx “Materials” tab in the import settings?

  2. Are there any animations involved? If yes, has the avatar been generated from the “Rig” tab in the import settings? 

  3. Is the origin/pivot set correctly? 

  4. Are the scales looking correct? 

  5. Are there any abnormal rotations? 

  6. Is the model named correctly?

    1. On Universal Naming Conventions  

  7. Is a prefab required? 

  8. Are any faces being removed due to ngons?
