VARLab : Rebasing in Sourcetree

Created by Jordan Robinson, last modified on Jul 06, 2022

In this article, we’ll cover how to do different types of rebasing in Sourcetree and explain when we would want to rebase a branch.

When we rebase a branch in Git, we’re changing the initial commit your branch was made from.

In this diagram you can see that the branch named Feature was originally based on the leftmost commit. After rebasing the Feature branch onto the commit that the Main branch is at, it’s as if the Feature branch was originally branched off the Main branch’s current commit.

Again, rebasing is changing the commit that your branch started from.

There are a few cases where it’s beneficial to know how to rebase your branch.

  • You’re about to create your pull request

  • You need a change that’s been merged into the branch you’ve branched off of

  • Your branch has been up for multiple days and you want to get ahead of any merge conflicts before they get out of hand

Standard Rebase

To begin rebasing your branch, make sure you have it checked out in Sourcetree. In this example we’ll demonstrate rebasing a trucking branch onto the current development branch commit.

With your branch checked out, it’s a good idea to Fetch just in case the branch you want to rebase onto has changes since your Sourcetree last updated.

Next we need to find the current commit that the development branch is on. You can do this by left-clicking the development branch in the REMOTES tab or just find it in the History.

Once you’ve found the correct commit you want to rebase onto, right-click it and select Rebase.

In this example we are assuming there are no merge conflicts during the rebase process. If you have conflicts that arise during your rebase, check out Merge Conflicts on how to handle them.

After the rebase is finished, you will see your branch’s commits stemming from the current development commit as if you had started your branch there all along.

Right now your branch has been rebased but only on your local machine. To update your branch on Bitbucket we need to push these changes but you’ll notice you also have changes to pull.

Because your local branch is now out of sync with the branch on Bitbucket, we need to use a force push to tell Bitbucket that our local version is the true version and we want to replace whatever’s on Bitbucket with what we have on our PC.

First we need to make sure force push is enabled in Sourcetree by going to Tools > Options > Git > Enable Force Push.

Now with the ability to force push enabled, click on Push and select the check box for Force Push.

After pushing, your rebase will be reflected on Bitbucket and your branch is ready for a pull request or any further development.

Interactive Rebase

An interactive rebase allows us to make edits to the commits themselves before rebasing. This means we can edit commit messages, delete commits, combine them, and a few other things.

Interactive rebases are great for cleaning up your branch’s Git history before you rebase onto development in preparation for your pull request.

To begin an interactive rebase on your branch, make sure you have it checked out in Sourcetree. In this example we’ll demonstrate using interactive rebase to squash commits and edit the final commit message.

Once your branch has been checked out, you will need to find the commit you wish to rebase onto. In the previous example we rebased on top of development but for this example we’re just going to squash the commits and edit their message.

It’s good to do your interactive rebase on top of the commit your branch was originally created from as merge conflicts will break an interactive rebase. That way you can squash your branch’s commits and edit the commit messages before finishing with a standard rebase onto development or whatever your target branch happens to be.

To find the commit your branch was created from, set the filter to Current Branch in Sourcetree when in the History tab and find the first commit that doesn’t have your ticket ID in its commit message. Once you’ve found it, right-click and choose to Rebase children of %commit% interactively…

Now you’ll be presented with a menu showing all the commits that exist on top of the one you chose as your rebase target.

Now if we want to squash these commits into one we can click on each one and select Squash with previous.

After squashing all the commits we can see we have a single commit but it has a really long message. We can click Edit Message to create a new message for this squashed commit.

Now we can hit OK to finish the rebase.

It didn't work!

If the rebase fails at this point you either have a merge conflict with the commit you chose to rebase on top of or some files in the rebase were locked by Unity, Visual Studio, Unity Hub, or Sourcetree.

If you see this error, a file is being locked.

To fix this error, quit out of Unity, Visual Studio, Unity Hub, and Sourcetree before trying again.

After the rebase has completed, you will see that you have commits to push and pull.

We need to tell Bitbucket that our version of the branch is correct so just like in the standard rebase we will need to use Force Push.

If you do not have the option to use force push, read the section on Standard Rebase above to see how to enable it in Sourcetree.

After pushing, your rebase will be reflected on Bitbucket.
