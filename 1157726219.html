VARLab : Security Testing(IN PROGRESS)

Created by Sana Javeed, last modified on Oct 10, 2024

It is necessary to guarantee the DLX platform’s dependability and stability through security testing.

Goals: To identify vulnerabilities & potential security threats and to then mitigate any risks involved through proposed proven solutions.

  • Verify the DLX implements proper authentication methods

    • Check if it protects user accounts

    • Check if it protects against unauthorized access, protects privacy and confidentiality of any sensitive data.

Sensitive data: Development console houses information like answer keys that should be NOT available on the production builds

Development data that is often logged to the “console” and would be visible to more tech-savvy users, or anyone who pressed F12 on their keyboard. This data should only be visible in “development”-level builds and should never be present in production.

SCORM uses the unique identifiers generated when the student gets added to econestoga that is used to retrieve the save / load details on the DLX. This exposes information like Student information email, passwords and student number.

LTI encodes student personal information (name, email, student number) as well as enrolment information (course code and so on) in the security token returned from authentication.

Cross Site Scripting (XSS) is enabled when user input on a web application is saved exactly as entered and then displayed again exactly as entered. This can even be a threat when the information is only shown to the user who originally entered the data.

The threat here is that the user input can include JavaScript that will be executed by the web browser when it tries to render the input. In testing we would use a script that pops up an Alert dialog but a real attack would include a malicious library that would install severe exploits on the user’s device.

In a single-user risk case the input text would be entered by an attacker looking for unattended computers and open sessions.

Cross Site Request Forgery attacks use a malicious link to cause an authenticated user to make an unwanted action. These attacks can be mitigated by use of per-session or per-request tokens generated by the server, and user confirmation dialogs on any significant action.
