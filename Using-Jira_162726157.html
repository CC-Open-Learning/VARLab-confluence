How To Name Tickets

Your first thought when it comes to naming your tickets is likely to describe the work you’re about to do.

For example, you would expect if you needed to add a checklist to a sim that tracks the student’s progress you could just write “Create checklist system to track progress” as your ticket name.

This describes what you’re doing but not why. In many cases “why” may be obvious but we want to write our :story: Story tickets in an empathetic way. What that means is we want to write our tickets describing who benefits from the work we’re doing, what problem we’re trying to solve with our solution, and why we need to solve it at all.

To give an example using the previous checklist ticket, we could write its summary like this:

“As a student I want to track my progress in the sim so I know what is done and what is left to do.”

The work for this ticket might be implementing a progress checklist but overall we know why we originally decided to do that work.

More Examples

It can be hard to think about tickets in this way but here are some more example tickets and how they could be rewritten.

Create system to localize strings

As a student I want to use the sim in my preferred language

Optimize sim for better performance

As a student I want the sim to run smoothly even on low-end hardware so I don’t need an expensive device to do my homework

Add fuel tank inspection

As a student, I need to inspect the fuel tanks on the exterior of the truck so that I can identify conditions that may make it unsafe to operate my vehicle

Sometimes You Are A Stakeholder

While most of our tickets are written through the lens of students or instructors, we as developers can also have empathetic tickets when the work done would affect us.

An example ticket would be the bug reporter we use in Unity that creates Jira tickets automatically, in this case it’s perfectly acceptable to write the ticket summary as:

As a dev I want to report bugs from inside Unity with stats/logs to speed up debugging

Another example would be the CORE Builder menu, it makes sense to write the :epic: Epic for that as:

As a dev I want to build for eConestoga quickly and consistently to speed up development

Tool creation and automation are important parts of the development process but the stakeholder is not a student or instructor, it’s all of us as developers.

When Not To Use Empathetic Tickets

Sometimes writing our ticket summaries in this way doesn’t really make sense or can create a lot of noise that makes it difficult to find tickets you’re looking for. Here are a few circumstances where you would describe the work you’re doing.

Tasks

Taking the examples above, if “As a student I want to use the sim in my preferred language“ were an :epic: Epic we would be creating tickets for each individual :task: Task and :story: Story within it.

If the ticket really doesn’t make sense to write in an empathetic form, it’s fine to make it a :task: Task where the summary is the work being completed.

Examples for :task: Tasks within this :epic: Epic would be:

“Create localized assets and strings for French language”

“Write Unit Tests for Localization system.”

Bugs

Another time to avoid using empathetic summaries is for Bug tickets. It is perfectly fine to explain the bug in your ticket summary rather than write it out in an empathetic form.

When it comes to bug ticket summaries, make sure to describe the effect the bug has on the sim rather than the technical reason the bug exists. We want to do this as it makes it easier to understand if we have a regression or check to see if a bug in a current build has been fixed in development.

Some examples of unhelpful bug ticket summaries are here on the left while the right would properly describe the bug.

Describes the solution (no context)

Describes the problem (provides context)

Nullref exception from brakechambers

Cannot progress past the brake chamber inspection

Move camera POI for windshield inspection

Windshield looks like it’s cracked when it isn’t due to PS Door Window crack showing through it

Modify tire shader to better show uneven wear

Difficult to see uneven wear during tire inspections

Review of Ticket Types

Depending on the size and type of the issue, it may qualify as an :epic: Epic , :story: Story, a :task: Task, or a Bug.

Ticket Type

When to use

:epic: Epic

Large, abstract features that will require many sub-tickets to complete.

Describes how the feature affects a stakeholder.

:story: Story

Feature that is estimated to take anywhere from 4-16 hours.

Describes how the feature affects a stakeholder.

:task: Task

Feature that is estimated to take anywhere from 4-16 hours.

Describes the work to be done.

Bug

Unintended behaviour is observed within the sim.

Describes the problem, not the work to be done.

No Local Knowledge

Every Jira ticket should express all the information required to understand what work was requested and what work was performed. This will help you, or anyone else, pick up where the ticket left off.

If you have completed a ticket and there’s information necessary to understand what you’ve done, add an explanation or some links in the description/comments so that people in the future can understand it.

This is especially helpful for situations where the information in your Jira issue isn’t really worth documenting on Confluence but may be useful to someone in the future nonetheless.

Statuses

As you work on your tickets, you will need to update the ticket status to reflect the state of the work within that ticket.

All tickets start in the TO DO status and move to IN PROGRESS when you begin working on it.

When you’ve completed your work for a ticket, it gets moved to IN REVIEW after which you can create a pull request for your branch.

Once your branch has been reviewed and the pull request is approved, the branch will be merged in and the ticket can be set to DEV COMPLETE.

The final status we use is DONE which occurs when your work is in the most current build of the sim available to the stakeholders.

If the ticket cannot be completed for reasons outside of your control, it can be marked as BLOCKED until someone is able to work on it again.

For tickets where the work is duplicated or is not something we decided to go ahead on for whatever reason, they get ARCHIVED.

Creating a Ticket

To create an :epic: Epic , :story: Story, a :task: Task, or Bug, navigate to the board the issue is for and select Create in the top bar.

In the Create issue popup you can select the project and issue type before filling out all the relevant information. For this example we have chosen an :epic: Epic.

For the rest of the fields, make sure you choose a relevant label and set the sprint to be the expected sprint the work will be for. If it’s for the current sprint, you may get a warning that you’re increasing the scope of work to be performed during the sprint.

Once you hit create, your issue has been made and you can make any sub-issues that you need. If an issue relates to yours, make sure to link them.
