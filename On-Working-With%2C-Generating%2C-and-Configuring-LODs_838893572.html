LODs

What is an LOD?

LOD stands for “Level of Detail”, and is a system used in games to try to optimize the performance of visuals. In general, an LOD system swaps out models or textures for lower fidelity versions of themselves, either based on the distance the object is from the camera, or based on what percentage of the screen the object takes up.

LODs are certainly not a new thing, being employed all the way back on the PS1 and N64

While the relative size of the object is the biggest factor at play here, and we rely on a smaller object having fewer pixels to display it and therefore less detail, it is a common strategy to also employ use of post processing to further hide the changes in the LODs

For our purposes we will often just two LODs, but it is not uncommon for a big budget game to have more than two. Depending on the size and importance of an object, as well as the number of angles a player can view it from a character or proper might go from being nearly cinema quality all the way to being reduced to a card or a standee - basically flat - before being culled entirely. Trees are a great example of an asset that will often have a card version of it before being culled.

Why Use LODs?

Great question! As mentioned above, an LOD system is an attempt to optimize the graphic processing available from a piece of hardware. While this is still a consideration today (and especially so for us as we deploy to web), these considerations were much more important with older hardware.

Lets look at the XBox 360 as an example. On paper, that console was supposed to be able to render 500 million polygons per second, but in reality that number is likely much lower, depending on the game. We can use 100 million triangles per second as an optimistic benchmark. With so few draw calls available, this would make a full forest like the Pine Forest in Skyrim impossible for the hardware to handle… without the use of LODs and other optimization techniques.

Be aware though, that all things come at a cost. While use of LODs certainly lessens the burden on a system's GPU the trade off is increased overhead on the CPU and RAM, depending on the setup. Games often don’t use the entirety of the CPU available to them, so this usually a moot point, but it is something to keep in mind.

LOD Naming Conventions

Like all our art assets, we want to have a descriptive and descriptive name. Objects or parts objects that use LODs are no exception, except for the exception that we don’t get to decide the convention for LODs. In our case, Unity did that for us.

GameObject_LOD0
GameObject_LOD1
GameObject_LOD2

Seen above is the naming convention that Unity expects for auto-detecting LODs. They're pretty simple. All Unity expects is that the very end of the part name has “_LOD[x]” where “X” is the level of detail.

LODs use a base 0 counting system, and the zeroth entry is the highest fidelity version, with each increment having less detail than the one that came before it.

These naming conventions, when present in an imported FBX signal to Unity that this game object should have the LOD Group Component. This component can be configured manually, but having proper names speeds this up quite a bit!

Working With LODs

LODs That Are Already There

When manipulating a model that uses LODs you should be careful to perform equal actions to all levels of detail in the model. If you don’t do this, the LOD system will not only be swapping the models and textures, but also the position, rotation, or scale when the camera moves closer or further:

There are two main things you can check to see if the model has LODs. The first is to check the hierarchy of the game object and check the names of the parts of the model to see if it has a name that includes “_LOD[x]” as described above. You can also check the top level of the game object to see if it has the LOD Group component

Configuring or Re-configuring LODs

Sometimes you may need to adjust an LOD Group component for one reason or another. The main reason would be that the swap between models is too obvious or jarring. The simplest way to fix this is to change the distance at which the models are swapped.

Using the control in the image above, you can drag the ends of the coloured sections of the bar in order set what distance (or screen percentage) the models are swapped at.

In the above picture we can see that LOD0 is active from 100% to 23% screen real estate, then LOD1 is active from 23% to 2% screen real estate. Finally the object is culled at 1% or less screen real estate.

Also note the bright blue camera icon. This represents the current position of the camera and shows what screen percentage the object takes up at that distance. You can use this in order to help gauge when the LODs should be swapped. You can click the “Set To Camera” buttons to move the break-point to the current camera position for the corresponding Level.

The other main reason to make an adjustment here is if a part needs to be added or removed. We can see the plus and minus buttons on each LOD. These can be used to add or remove meshes as needed. Note also the “Recalculate Bounds” button. This button will find the outer bounds of the meshes involved in this LOD group, and will set the bounds accordingly. These bounds are used to gauge what screen percentage the LOD Group is taking up.

LOD Heirarchy

We can avoid some issues that can arise when working with LODs by setting a deliberate heirarchy to our models. It’s possible that a model may be received or retrieved with the LODs arranged differently from what will help us out. These will usually come from historical assets, or assets that we have purchased or otherwise received from outside of VAR Staff generating them. Don’t worry though, this hierarchy can be set up in engine!

We will aim to have a hierarchy that places a lower detail LOD as a direct child of the previous level of detail.

Sometime we will have more complex models that will have sub-meshes, and each could have its own LOD. It could also mean that some of the sub meshes do NOT have their own LODIn this case, we will want to maintain a proper model hierarchy as well as the LOD hierarchy, which would look something like this:

In the above picture we can see that the model’s overall hierarchy is:

  • Door Frame

    • Latch

    • Door

      • Gasket

      • Handle

      • Spring

Adding in the LODs we would instead see:

  • Door Frame LOD0

    • Door Frame LOD1

    • Latch LOD0

      • Latch LOD1

    • Door LOD0

      • Door LOD1

      • Gasket LOD0

        • Gasket LOD1

      • Handle LOD0

        • Handle LOD1

      • Spring LOD0

        • Spring LOD1

Note that DoorFrame LOD1 and Latch LOD0 are on the exact same level. DoorFrame LOD1 is directly a child of its higher fidelity LOD, and Latch is still a child of the DoorFrame as it was in the previous example.
